In this module you will launch a cloud development environment, make a code change with the help of an AI coding assistant, push your change, and watch it flow through the automated CI/CD pipeline.

== Launch Dev Spaces

. In the *OpenShift Console*, switch to the *Developer* perspective and make sure you are in the `{dev_ns}` namespace

. Click *Topology* in the left sidebar

. On the Parasol Insurance deployment, click the *Dev Spaces icon* (the icon that looks like a code editor, with the tooltip "Edit source code")
+
This opens your cloud development environment using Red Hat OpenShift Dev Spaces. The workspace is defined by a `devfile.yaml` in the repository — it specifies the container image, tools, extensions, and runtime configuration.

. When prompted, accept the GitLab OAuth authorization popups

. If prompted to log in, enter your credentials:
+
* *Username:* `{user}`
* *Password:* `{password}`

. Wait for the workspace to start. This may take a minute or two on the first launch as the container image is pulled and the environment is configured.

== Explore the IDE

Once the workspace loads, take a moment to look around:

* *Explorer* (left sidebar): Shows the `parasol-insurance` project structure — a standard Quarkus application
* *Extensions*: In the left sidebar, click the Extensions icon to see pre-installed extensions:
** *RHDA* (Red Hat Dependency Analytics) — scans your dependencies for known vulnerabilities
** *Roo* and *Continue* — AI coding assistants that can help you write and refactor code

== Start the Application in Dev Mode

. In the top menu, click *Terminal > Run Task...*

. Select *devfile > Start Dev Mode*
+
This starts Quarkus in development mode. Behind the scenes, Testcontainers spins up local Kafka and PostgreSQL instances so the full application runs inside your workspace.

. Wait for the application to start. You will see log output in the terminal indicating the Quarkus application is ready.

. When prompted, open the application preview in a new browser tab. Navigate to the *Inbox* page to confirm the application is running and displaying emails.
+
[NOTE]
====
Curious about how nested containers work in Dev Spaces? Open a terminal and run `podman ps` to see the Kafka and PostgreSQL containers running alongside your application.
====

== Make a Code Change with Claude Code

Now you will use Claude Code, an AI coding assistant available in the terminal, to make a code change.

The inbox currently displays timestamps in a UTC-like 24-hour format (e.g., `2026-04-20 14:30:00`). The product requirements call for a locale-friendly format with AM/PM (e.g., `Apr 20, 2026 02:30 PM`).

. Open a new terminal in Dev Spaces (*Terminal > New Terminal*)

. Type `claude` and press Enter to launch Claude Code

. Enter the following prompt:
+
[source,text]
----
the inbox currently displays utc-like time, but the product requirements need a locale time with AM/PM - please implement this change
----

. Claude Code will analyze the codebase, find the relevant code (the `DateTimeFormatter` pattern in `EmailRouter.java`), and propose changes. Review the suggested changes and accept them.

. After accepting the changes, verify them in your running application:
+
* Switch to the browser tab with the Parasol app preview
* Refresh the *Inbox* page
* Confirm the timestamps now display in a locale-friendly format with AM/PM (e.g., `Apr 20, 2026 02:30 PM`)

== Commit and Push

. In the Dev Spaces left sidebar, click the *Source Control* icon (the branch icon)

. Review the changed files in the *Changes* section

. Stage your changes by clicking the *+* icon next to each changed file (or click *+* next to *Changes* to stage all)

. Enter a commit message, for example: `feat: display inbox timestamps in locale format with AM/PM`

. Click the checkmark icon to commit

. Push your changes by opening a terminal and running:
+
[source,bash,role=execute]
----
git push
----

== Watch the Pipeline

Your `git push` triggered a webhook in GitLab, which started a new Tekton PipelineRun in your build namespace.

. Switch to the *OpenShift Console*

. Navigate to the `{build_ns}` namespace

. Switch to the *Developer* perspective and click *Pipelines* in the left sidebar

. You should see a new PipelineRun in progress. Click on it to watch the stages:
+
* *clone* — pulls your latest code from GitLab
* *maven-build* — compiles the application and runs unit tests
* *sonar-scan* — runs SonarQube static analysis
* *build-and-push* — builds a new container image and pushes it to Quay
* *rollout-restart* — triggers a rolling restart in the dev namespace

. Wait for the pipeline to complete. This typically takes 3-5 minutes.

== Check the SonarQube Report

. In the PipelineRun details, click on the *sonar-scan* task

. Expand the task logs and look for the line that reads:
+
----
SonarQube Report: <URL>
----

. Copy and open that URL in the *SonarQube* tab on the right side of this page to see the analysis report for your project. The report shows code quality metrics, potential issues, and test coverage information.

== Verify the Deployment

. Switch to the `{dev_ns}` namespace in the *OpenShift Console*

. In the *Developer* perspective, click *Topology*

. Confirm the Parasol Insurance pod has rolled out with the new image (you may see a brief rolling deployment)

. Click the route icon on the application to open it

. Navigate to the *Inbox* page and confirm the timestamps now display in the locale format with AM/PM

Congratulations — you have made a code change with an AI coding assistant, pushed it through an automated CI/CD pipeline with quality gates, and verified the deployment. In the next module, you will enhance the application with LLM-powered email classification.
