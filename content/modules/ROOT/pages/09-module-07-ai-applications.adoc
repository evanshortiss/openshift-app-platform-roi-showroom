In Module 1 you saw that the Parasol Insurance inbox uses simple keyword matching to classify emails. Ambiguous emails that the keywords cannot confidently classify are flagged as *REVIEW REQUIRED*, requiring manual triage by a claims administrator. In this module you will replace that keyword matching with an LLM-powered classification service that intelligently routes every email.

== Overlay the LLM Classification Code

The LLM classification code has been prepared on a branch in your repository. You will overlay it onto your current working branch.

. Return to your *Dev Spaces* workspace (reopen it from the Topology view if needed)

. If Quarkus dev mode is still running, stop it by pressing `Ctrl+C` in the terminal where it is running

. Open a new terminal and run the following command to overlay the LLM classification code:
+
[source,bash,role=execute]
----
git checkout origin/llm-classification -- .
----
+
This pulls the LLM classification files onto your current branch without switching branches.

== Review the Changes

Take a few minutes to review the key files that were added or modified:

. Open `src/main/java/com/parasol/EmailClassifier.java`
+
This is the LLM classification service. Notice the `@RegisterAiService` annotation from Langchain4j and the `@SystemMessage` that defines the classification rules. The system message tells the LLM how to categorize each email — what qualifies as a claim, a follow-up, a new customer inquiry, etc.

. Open `src/main/java/com/parasol/EmailRouter.java`
+
Previously this file contained the keyword matching logic. Now it delegates classification to the `EmailClassifier`. The router sends each email's content to the LLM and uses the response to route the email to the appropriate category.

. Open `src/main/resources/application.properties`
+
Look for the LLM configuration properties:
+
----
%dev.quarkus.langchain4j.openai.base-url=${LLM_BASE_URL}
%dev.quarkus.langchain4j.openai.api-key=${LLM_API_KEY}
----
+
The `%dev` prefix means these properties apply when running in Quarkus dev mode. The `LLM_BASE_URL` and `LLM_API_KEY` environment variables are already configured in your Dev Spaces workspace.
+
In the deployed environments (dev and prod namespaces), these credentials are injected by the ExternalSecrets you saw in Module 1 — the platform engineer configured Vault to provide LLM credentials, and custom Dev Spaces base images can include additional tools and configuration.

== Test the LLM Classification

. Start the application in dev mode again using *Terminal > Run Task... > devfile > Start Dev Mode*

. Wait for Quarkus to start and open the application preview

. Navigate to the *Inbox* page

. Notice the difference: there are *no REVIEW REQUIRED* emails. Every email has been intelligently classified with a specific category and a detailed routing reason from the LLM.
+
Compare this to what you saw in Module 1 where several emails were flagged for manual review. The LLM understands the content and context of each email, making nuanced classification decisions that keyword matching cannot.

. Wait for the pipeline to complete.

== Promote to Production

So far, all your changes have deployed to the dev environment. Now you will promote the LLM classification feature to production using a tag-based GitOps workflow.

. In the Dev Spaces terminal, create and push a Git tag:
+
[source,bash,role=execute]
----
git tag 1.0 && git push --tags
----

. Switch to the *OpenShift Console* and navigate to the `{build_ns}` namespace

. A new PipelineRun starts — this is the *tag pipeline* (`parasol-insurance-tag-promote`). It builds a tagged image and then runs an `update-prod-manifest` task that creates a merge request in your `parasol-insurance-manifests` repository to update `values-prod.yaml` with the tag `1.0`.

. Wait for the tag pipeline to complete.

== Merge the Production Promotion

. Open *GitLab* (use the tab on the right, or navigate to {gitlab_url})

. Navigate to your `parasol-insurance-manifests` repository

. Click *Merge Requests* in the left sidebar

. Open the merge request created by the tag pipeline. Review the changes — it updates `image.tag` from `latest` to `1.0` in `values-prod.yaml`.

. Click *Merge* to merge the changes

== Verify the Production Deployment

. Switch to the *OpenShift Console*

. Navigate to the `{prod_ns}` namespace

. In the *Developer* perspective, click *Topology*

. Watch the Parasol Insurance pod update with the new production image

. Once the rollout completes, click the route icon to open the production application and confirm the LLM classification is working

== Summary

You replaced a naive keyword classifier with an LLM-powered classification service, deployed it through the same trusted CI/CD pipeline, and promoted it to production using a tag-based GitOps workflow. The LLM credentials were provided by the platform through ExternalSecrets and Vault — you never had to manage API keys manually.

The same pipeline, the same security controls, the same GitOps delivery — whether you are changing a date format or integrating an LLM, the platform handles the complexity so you can focus on the code.
